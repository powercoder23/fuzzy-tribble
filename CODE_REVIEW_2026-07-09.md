# Project Review — 2026-07-09 (trader + senior architect pass)

Scope: full flow (compose → collector → scanners → gates → OrderManager → paper book → dashboard), trading logic, missing links, dashboard essentials. Two findings were verified by executing the actual code, not just reading it.

---

## 1. Verdict

The architecture is sound and unusually disciplined for a solo project: single-writer SQLite contract, WAL everywhere, zero-API scanners, fail-open gates with loud alerts, honest-economics paper P&L (spread + full NSE fee schedule), factor snapshots per trade for edge attribution, and a separate monitor thread so scans can't delay an SL. The P0/P1 review items were genuinely fixed, not just documented.

The problems that remain are concentrated in three places: **(a) one real state-machine bug that affects every Break & Bounce paper trade, (b) a staleness hole in the Sonar entry veto, and (c) the absence of any book-level daily-loss guard** — the one risk control a trader would demand first.

---

## 2. Confirmed bugs (verified by execution)

### BUG-1 — Full-booked single-target trades never close (hits 100% of B&B trades)
`paper_trader.apply_tick`: B&B books via `submit_external_signal` with `t1 == t2` and `t1_book_fraction = 1.0`. When the target hits, Phase 1 books 100% at T1 and sets `t1_done`, but never finalizes; Phase 2 is skipped because `qty_frac == 0`. Verified:

```
apply_tick(trade{entry=100, t1=t2=125, frac=1.0}, 126)
→ events ['T1'], status 'open', qty_frac 0.0, exit_reason None
... stays a phantom open position until 15:20 →
→ events ['TIME'], reason 'Time 15:20', realized 25%
```

P&L ends up correct, but until square-off the zero-quantity trade:
- occupies a Gate-5 `MAX_SIMULTANEOUS` slot and the direction/sector concentration caps → blocks real entries all afternoon;
- costs one full option-chain fetch per 5-min monitor tick (at 3.1s/req pacing this is real API budget);
- sends the wrong alert ("T1 hit — booked partial" on a full exit) and records wrong `exit_reason`/`closed_at`.

**Fix (small):** in Phase 1 after the T1 booking, `if trade["qty_frac"] <= 1e-9: _finalize(trade, "T1 full"); events.append("T2"); return events`. Add a unit test with `t1 == t2, frac = 1.0`.

### BUG-2 — Sonar entry veto uses stale (previous-session) signals
`paper_trader.process_signals` vetoes on `get_latest_sonar(sec_id)` with **no timestamp check**. `get_latest_sonar` returns the latest row ever. So at 09:30, yesterday's 15:25 `FLAT`/`BREAKDOWN` rows veto today's discount entries until sonar has written fresh rows — and if the sonar container is down, every veto for the whole day comes from the last session.

The codebase already knows this is wrong: `OrderManager._check_position_risks` explicitly discards sonar rows whose `timestamp[:10] != today`. The entry veto needs the same filter (same-day, ideally also a max-age in minutes, like `entry_gate.ENGINE_MAX_AGE_MIN` does for the engine).

Related inconsistency worth a deliberate decision: a symbol with **no sonar coverage passes** the veto while a symbol with a `FLAT` reading is blocked. No-data and no-trend get opposite treatment.

## 3. Missing links — trader's lens

### 3.1 No book-level daily-loss limit (the biggest gap)
Nothing stops the shared paper book from taking its full 5 trades and 5 full stop-losses every single day. Per-trade rupee risk cap exists (discount only), trade-count caps exist, but there is **no "down ₹X today → stop entering"** guard. Worse, B&B's `daily_loss_limit_pct` is dead code in paper mode: `record_trade(pnl)` is only ever called at entry with `pnl=0`, and exits happen inside `paper_trader`, so `daily_pnl` never moves.

**Fix:** in `OrderManager.submit_signals` / `submit_external_signal`, sum today's `realized_rupees` from the book (+ open-trade marked P&L if you want to be strict) and block new entries below a configurable floor. Surface it as a Settings flag and a dashboard meter. This matters even in paper mode — otherwise the paper results you're using to green-light live trading were generated by a system with a risk profile you'd never run live.

### 3.2 Cap asymmetry between B&B and discount
`book.count_today()` counts **all** rows for the date, so every B&B booking silently consumes the discount's `max_signals_per_day` budget — while B&B itself ignores that cap. On a busy breakout day B&B can starve the discount strategy entirely. Either count the cap per strategy tag, or rename/document it as a global book cap. Same story for Gate-5 `MAX_SIMULTANEOUS` (BUG-1's phantom opens make it worse).

### 3.3 Live-order path contradicts the architecture doc
CLAUDE.md says "Upstox data-only, Dhan reserved for execution" — but `upstox_adapter.place_order` is a fully implemented **live Upstox execution path**, and `AUTO_EXECUTE=true` would route B&B/momentum bracket orders through it. Before any live switch, three things are missing:
- **Persistence**: B&B `open_positions`/`trades_today` are in-memory; a container restart mid-session orphans live positions (its `_force_exit_all_positions` iterates the in-memory list).
- **Reconciliation**: no startup sync against the broker order book / positions API.
- **Kill switch**: no global "flatten everything now" control (Settings toggle + endpoint).
Until those exist, keep `AUTO_EXECUTE` physically absent from `.env` (it currently is — good).

### 3.4 Costs silently skipped when lot_size falls back to 1
`_finalize` gates the fee model on `lot > 1`. If the lot sizer fails (it falls back to `lambda _s: 1` in `main.py`), every trade that day books **zero costs and zero slippage-rupees**, inflating paper P&L exactly when infrastructure is degraded. Use an explicit sentinel (e.g. `lot_size=None` in unit tests) instead of overloading `1`.

### 3.5 Fixed-% premium exits ignore the instrument
SL −15% / T1 +25% / T2 +45% of premium is applied identically to a 0.15-delta 20 DTE option and a 0.40-delta 6 DTE option. Premium noise alone (spread fallback is 2%) trips a 15% stop on low-priced contracts. You have IV, delta, and expected-move per candidate at entry — vol-scaling the SL (or anchoring it to an underlying level, as B&B does natively) is the single highest-leverage strategy improvement available. The factors_json capture means you can back-test this from your own history.

### 3.6 Smaller items
- `NSE_HOLIDAYS = []` — the 2026 list was never populated; DTE and expiry filters degrade to weekend-only (off by 1 around every exchange holiday).
- Scan jobs run 09:15–15:15 but entries stop at 14:00 — the 14:00–15:15 full-universe sweeps (with chain fetches) only feed the suggester. Consider a lighter post-cutoff cadence.
- Dedup (`has_trade_today` on symbol+strike+side) permanently blocks same-day re-entry after a stop-out — probably intended, but note that a fresh, valid afternoon signal on the same strike is unreachable.
- Monitor re-prices each open trade with a **full chain fetch per trade**; trades sharing an underlying+expiry could share one fetch.
- `.env` is git-ignored (verified) but contains a live Discord webhook and PINs — rotate the webhook if this folder is ever shared/synced anywhere.

## 4. Architecture assessment

What's right (keep):
- Sole-writer contract + WAL + busy_timeout via `iv_store.connect()` — correct SQLite multi-container pattern.
- Gate stack is layered sensibly: structural quality (pre-market 5-gate) → tape (breadth) → conviction (composite/engine) → concentration → universal guards in `paper_trader`. Fail-open with a once-per-day loud alert is the right trade-off for a paper system.
- OrderManager as the single owner of the position lifecycle, with B&B routed through `submit_external_signal` — the strangler path to V2 is clean, and `entry_gate.GATE_SOURCE="engine"` gives you the flip switch already.
- Honest economics (gap-aware stop fills, spread slippage, full fee schedule) — most retail systems never do this.

Watch items:
- `discount.py` at 3,170 lines remains the god module (known, on the backlog) — `scan_underlying`, chain client, execution-price logic, and Telegram all live there; every service imports it, which is why sonar needs the fat Docker image.
- Config sprawl: 12+ `*_config.py` files plus env vars plus the settings-DB flag registry, three override layers deep (`settings_store` → env → config). It works, but the precedence is only documented in docstrings; one `CONFIG.md` table would prevent a future foot-gun.
- Gates read the settings DB per candidate (`pre_market_gate.evaluate` → `flag_str` each call); the 60s flag cache makes this fine — just don't drop the cache.

## 5. Dashboard — essentials check

Present and good: KPI strip (net/win-rate/expectancy/PF), cumulative P&L history, per-strategy performance, opportunities (with honest `as_of` staleness), market snapshot with **nulls never faked**, sector trend, activity feed, IV/skew/decay/tilt analytics, convex journal + regime cockpit, health, settings with flag registry + container control.

Missing for a trading cockpit (priority order):

1. **Open-positions panel with live unrealized P&L** — distance to SL/T1 per open trade and an aggregate open-risk number. The data is already in `paper_trades` (`last_price` updates every 5 min); it's a frontend + one endpoint aggregation away.
2. **Daily loss meter** — today's realized + marked P&L vs the (to-be-built, §3.1) daily-loss cap. This is the number a trader looks at 20 times a day.
3. **Entry-funnel / rejection panel for the live paper path** — today: candidates scanned → dropped by pre-market gate / breadth / concentration / sonar veto / risk cap → booked. All of it currently exists only in logs. The convex journal has exactly this for the engine; the money path deserves the same.
4. **MAE/MFE per trade** — record running max/min of `last_price` in `monitor()` (two columns, trivial write) and chart it. This is the data that tells you whether −15%/+25% are the right levels; without it §3.5 stays guesswork.
5. **Frictions panel** — gross vs net, cumulative costs + slippage (columns already exist per trade).
6. **Cap/limit status strip** — trades used today (n/cap), per-symbol caps hit, direction/sector concentration state.
7. **Data-freshness banner on the main page** — age of the last `iv_history` snapshot (health endpoint already computes it; surface it globally, since every gate silently depends on it).
